## Instructions

Reason step by step:
1. What exact error appears in the logs? Which class/method/module threw it?
2. Check if this error pattern matches any past incident in the knowledge base above. If so, reference the prior incident ID and note that this is a recurring pattern.
3. Is there a blame PR? If yes, identify which line(s) in the PR diff introduced this error. If no, determine whether this is a Kubernetes issue, network issue, TLS/certificate issue, or platform issue based on the log patterns and alert metrics.
4. Why does the identified change or infrastructure condition cause the error? (e.g., JPA entity field removed but Flyway migration not added, unbounded cache causing OOM, N+1 queries exhausting connection pool, XSUAA config mismatch, CDS feature flag enabling unbounded deep expand, pod memory limits too low for traffic, expired TLS certificate, network policy blocking DNS resolution after cluster upgrade)
5. What is the blast radius — which endpoints, users, or BTP services are affected?
6. Choose the most appropriate remediation type:
   - `flyway_migration`: DB schema out of sync — a SQL migration is needed
   - `entity_rollback`: A JPA entity field needs to be restored with correct annotations
   - `config_fix`: A config file (application.yml, xs-security.json, mta.yaml) needs updating
   - `code_fix`: A specific Java/Node.js method or class needs to be fixed
   - `feature_flag`: A feature toggle in config (cds.features in package.json, Spring Boot feature flags in application.yml) should be disabled to immediately stop the blast radius — prefer this when a new feature flag was the root cause and disabling it is safer than a code fix
   - `pr_rollback`: Revert the entire blame PR — prefer this for P1/P0 incidents where the change is too broad or risky to forward-fix quickly, and rolling back is the fastest path to restoring service
   - `infrastructure_action`: Infrastructure requires a configuration change (Kubernetes manifests, Helm values, network policies, cert-manager resources, HPA settings) AND a GitHub Actions workflow to apply it. Choose this when there is no blame PR and the fix is an infrastructure configuration change.
   - `escalation`: The incident requires human or external intervention that cannot be expressed as a code or infrastructure config change — e.g., cloud provider support ticket, manual hardware intervention, external DNS provider action, or the root cause is unclear and needs expert investigation.
