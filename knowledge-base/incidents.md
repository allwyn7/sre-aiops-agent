## INC-2023-017 — 2023-11-15

**Title:** order-srv: HTTP 500 spike – Flyway migration version conflict after hotfix merge
**Severity:** P2 | **Service:** `order-srv`

### Pattern
`org.flywaydb.core.api.FlywayException: Found more than one migration with version`

### Root Cause
Two developers independently created V12 Flyway migrations on separate branches. When both branches merged to main, the application failed to start because Flyway detected duplicate migration version V12. The CI pipeline did not check for version collisions across branches.

### Resolution
Renamed one migration to V13 and added a CI check (`ci.yml`) that validates Flyway migration version uniqueness before merge. Added a team convention to always check the highest migration version on main before creating a new one.

**Post-Incident Issue:** https://github.com/sap-internal-demo/order-srv/issues/89
**Fix PR:** https://github.com/sap-internal-demo/order-srv/pull/91

## INC-2023-022 — 2023-12-03

**Title:** inventory-srv: connection pool exhaustion – missing @Transactional causing leaked connections
**Severity:** P1 | **Service:** `inventory-srv`

### Pattern
`java.sql.SQLTransientConnectionException: HikariPool-1 - Connection is not available, request timed out after 30000ms`

### Root Cause
PR #134 added a new batch import endpoint that performed multiple JPA operations without a `@Transactional` annotation. Each operation acquired a separate connection from the pool but Hibernate's auto-commit mode kept connections open longer than expected. Under load (50+ concurrent batch imports), the 10-connection HikariCP pool was exhausted within minutes.

### Resolution
Added `@Transactional` to the batch import method so all DB operations share a single connection per request. Increased HikariCP `maximum-pool-size` from 10 to 20 as a safety margin. Added a Prometheus alert for `hikaricp_connections_pending > 5` to detect pool pressure early.

**Post-Incident Issue:** https://github.com/sap-internal-demo/inventory-srv/issues/156
**Fix PR:** https://github.com/sap-internal-demo/inventory-srv/pull/158

## INC-2024-009 — 2024-02-20

**Title:** bookshop-srv: HTTP 403 for all service-to-service calls after XSUAA plan migration
**Severity:** P1 | **Service:** `bookshop-srv`

### Pattern
`com.sap.cloud.security.xsuaa.token.TokenValidationException: JWT token validation failed: audience validation error`

### Root Cause
The BTP platform team migrated the XSUAA service instance from `application` plan to `broker` plan. This changed the `xsappname` from `bookshop-prod` to `bookshop-prod!b12345` (broker plan appends `!bNNNNN` suffix). The `xs-security.json` in the repo was updated, but the XSUAA service binding in CF was not re-created, so existing JWTs still contained the old audience claim. All service-to-service calls from order-srv and inventory-srv started failing with 403.

### Resolution
Re-created the XSUAA service binding and restaged the application. Updated the deployment runbook to include "re-bind XSUAA after plan migration" as a mandatory step. Added a startup health check that validates XSUAA audience configuration matches the bound service instance.

**Post-Incident Issue:** https://github.com/sap-internal-demo/bookshop-srv/issues/203
**Fix PR:** https://github.com/sap-internal-demo/bookshop-srv/pull/205

## INC-2024-012 — 2024-03-08

**Title:** bookshop-cap-srv: OData requests timeout after enabling unbounded `$expand` feature flag
**Severity:** P2 | **Service:** `bookshop-cap-srv`

### Pattern
`[cds] - Error: HANA connection queue depth exceeded` combined with `deep_reads enabled: expanding associations recursively`

### Root Cause
PR #58 set `"deep_reads": true` under `cds.features` in `package.json`. With no depth limit configured, the CDS OData handler recursively expanded all associations for every row in the result set — generating 300+ HANA queries per single OData request. The 30-connection HANA pool was exhausted within seconds under normal production load.

### Resolution
Set `"deep_reads": false` in `package.json` under `cds.features` to immediately restore service. A proper forward-fix can re-enable deep reads with explicit `$expand` depth limits and query complexity guards.

**Post-Incident Issue:** https://github.tools.sap/I527229/sre-aiops-agent-hackathon/issues/12
**Fix PR:** https://github.tools.sap/I527229/sre-aiops-agent-hackathon/pull/14

## INC-2024-002 — 2024-04-10

**Title:** bookshop-srv: HTTP 500 spike – Hibernate SQLGrammarException on Books endpoint
**Severity:** P1 | **Service:** `bookshop-srv`

### Pattern
`org.postgresql.util.PSQLException: ERROR: column does not exist`

### Root Cause
PR #52 removed the 'price_old' field from Book.java, but no corresponding Flyway migration was introduced to remove the column from the PostgreSQL schema. This caused Hibernate to attempt SELECT queries that still reference 'price_old', resulting in SQLGrammarException and PSQLException: ERROR: column 'price_old' does not exist.

### Resolution
Add a Flyway migration to drop the column from PostgreSQL that corresponds to the removed JPA entity field. Ensure that schema changes are consistently applied during deployments.

**Post-Incident Issue:** https://github.com/allwyn7/sre-aiops-agent/issues/3
**Fix PR:** https://github.com/allwyn7/sre-aiops-agent/pull/4
