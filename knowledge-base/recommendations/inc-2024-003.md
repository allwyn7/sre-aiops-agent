## Disaster Recovery Assessment

### Single Points of Failure
- Single database without cluster mode or replica setup.
- Endpoint '/api/books' as single point of load during traffic spikes.

### High Availability Recommendations
Set up database replication within SAP HANA for failover.,Enable an in-memory cache (e.g., Redis or Hazelcast) to reduce excessive database reads for high-traffic endpoints.,Implement circuit breakers in the Spring Boot application to fallback gracefully under outages.

### Failover Strategy
Introduce secondary database replicas for read-heavy workloads and configure application failover using service discovery mechanisms provided by SAP BTP.

### RTO / RPO Assessment
Target RTO: 15 minutes. RPO: Near-zero due to transactional nature and high SLA service requirements. Real-time replication and monitoring must ensure these targets are met.

### Backup Recommendations
Increase data backup frequency and ensure backups are stored in a geographically redundant location. Include automated, periodic restore tests.

## Performance Tuning

### JVM Tuning
Use JVM options: -Xms256m -Xmx512m for this service. Use G1 Garbage Collector for predictable pause times for larger memory pools: -XX:+UseG1GC.

### Spring Boot Configuration
Set server.tomcat.max-threads: 200 and spring.datasource.tomcat.max-wait: 30000 in application.yml.

### JPA / Hibernate Tuning
Set `hibernate.default_batch_fetch_size=50` in the application's persistence configuration.

### Database (PostgreSQL) Tuning
Evaluate increasing the SAP HANA connection limit and optimize indices around the 'author_id' field in the book table.
