# SRE Recommendations: INC-2024-002

> **Service:** `bookshop-srv` | **Severity:** P1 | **Date:** 2024-04-10

---

## Capacity Planning

### Current Utilization
During the incident, HTTP 5xx errors spiked to 96% on affected endpoints with 91 errors per minute. Database connection pool maxed at 10 connections.

### Scaling Thresholds
Scale out when HTTP requests exceed 70 requests/min to avoid exceeding database connection pool limits. Increase HikariCP max pool size beyond 10 if consistent throughput exceeds current capacity.

### Resource Recommendations
1. Increase HikariCP pool size to 20. 2. Adjust Kubernetes resource limits: CPU `500m` -> `1000m`, Memory `512Mi` -> `1024Mi`. 3. Add a secondary replica for better load distribution across pods.

### Growth Projections
At 2x current load, the database pool and resources are likely to saturate. Recommend 2 replicas with HPA scaling configured for `cpu_utilization=70%`. At 5x load, move to read replicas or connection multiplexing via PgBouncer.

### Cost Considerations
Over-provision by scaling CPU/mem to avoid downtime at current traffic + 30%. Evaluate costs of read replicas for DB vs incident risk mitigation.

## Disaster Recovery Assessment

### Single Points of Failure
- Database table schema not validated before production deployment.
- Database migrations are coupled with application deployments, causing mismatches.

### High Availability Recommendations
Introduce a read replica for the database, implement Flyway schema validations, and decouple schema migrations from application release cycles.

### Failover Strategy
For database failure, introduce database failover with a standby replica; for application failure, use Kubernetes HPA or manual pod scaling to horizontally scale up.

### RTO / RPO Assessment
RTO: <30 mins (dependent on schema fix time). RPO: 5 mins with regular DB backups or WAL replication.

### Backup Recommendations
Implement automated multi-region backups to meet the 5-minute RPO goal. Test recovery from backups quarterly to ensure integrity.

## Performance Tuning

### JVM Tuning
Set JVM options: `-Xmx1024m -Xms512m -XX:+UseG1GC`. Utilize G1GC for predictable pause times.

### Spring Boot Configuration
Modify `application.yml`: `server.tomcat.max-connections: 200`, `server.tomcat.accept-count: 100`.

### JPA / Hibernate Tuning
Optimize Hibernate batch processing: `hibernate.jdbc.batch_size=20`, `hibernate.order_inserts=true`, `hibernate.order_updates=true`. Reduce lazy-loading for critical entities.

### Database (PostgreSQL) Tuning
Increase `max_connections` in PostgreSQL or consider PgBouncer. Adjust `shared_buffers` to 25% of system memory and `work_mem` to 4MB for better query handling.

### Connection Pool (HikariCP) Tuning
Increase HikariCP settings: `maximumPoolSize=20`, reduce `idleTimeout=30 seconds` to minimize lingering unused connections, enable `leakDetectionThreshold=10000ms`.

### Kubernetes Tuning

Set HPA target for pod CPU utilization at 70%. Configure `readinessProbe` to check DB connectivity. Increase `memory: 1024Mi` and `cpu: 1000m` per pod.

### Network Tuning

Set a reasonable connect/retry timeout for external services using `spring.cloud.gateway.httpclient.connect-timeout=5000ms` and `read-timeout=10000ms`.
