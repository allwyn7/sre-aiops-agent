## Capacity Planning

### Current Utilization
HTTP 5xx rate spiked to 91 per minute compared to a normal baseline of 0.2 per minute. 96.1% error rate indicates a system-wide failure during the incident period.

### Scaling Thresholds
Core scaling thresholds:
- HTTP 5xx rate crossing 5 per minute.
- HikariCP pool saturation 85% (e.g., if connection pool usage exceeds 8/10).
- Significant increase in database query response times (e.g., >500ms average response time).

### Resource Recommendations
1. Increase HikariCP pool size from 10 to 15.
2. Allocate more heap to the JVM (e.g. increase from 256MB to 512MB -Xms and -Xmx).
3. Add a Horizontal Pod Autoscaler (HPA) with minimum 2 pods and maximum 6 replicas based on CPU utilization > 70%.

### Growth Projections
At 2x load: HikariCP pool size may need to be increased to 30.
At 5x load: Consider sharding the database or enabling connection scaling in your DB.
At 10x load: Require multi-region replication, significant database scaling, and caching mechanisms.

### Cost Considerations
Over-provisioning resources (e.g., doubling Hikari CP size) would lead to increased infrastructure costs but reduce the risk of downtime during peak load. Evaluate predicted traffic growth and adjust capacity accordingly.

## Disaster Recovery Assessment

### Single Points of Failure
- PostgreSQL as a single point of failure for database operations.

### High Availability Recommendations
Deploy PostgreSQL in a high availability configuration using replicas and failover. Use AWS RDS Aurora or equivalent managed database for automated failover.

### Failover Strategy
Configure application database connection URLs with multiple hosts and a load balancer for DB failover. Utilize Spring Boot’s failover properties to retry database queries if one node fails.

### RTO / RPO Assessment
Recovery Time Objective: 10-30 minutes
Recovery Point Objective: Minimal data loss if using periodic backups and WAL archiving.

### Backup Recommendations
Implement daily full backups and real-time WAL archiving for PostgreSQL. Use automated restore testing to validate.

## Performance Tuning

### JVM Tuning
Increase heap allocation from -Xms256m -Xmx256m to -Xms512m -Xmx512m for improved memory management. Consider enabling G1GC with `-XX:+UseG1GC` for better handling of high loads.

### Spring Boot Configuration
Set 'server.error.include-message=request' in application.yml for clearer error reporting. Validate maximum database connection properties handling failures gracefully.

### JPA / Hibernate Tuning
Enable SQL statement batching; add 'hibernate.jdbc.batch_size=20' in application.yml to optimize database interactions.

### Database (PostgreSQL) Tuning
Set PostgreSQL shared_buffers to 25% of instance memory (e.g., 1GB on a 4GB system). Increase work_mem to 32MB for query operations.

### Connection Pool (HikariCP) Tuning
Increase HikariCP pool size from 10 to 15. Enable leak detection by setting 'leakDetectionThreshold' to 5000ms in application.yml.

### Kubernetes Tuning

Set initial pod resource requests/limits to 300m CPU and 512Mi memory. Add an HPA to maintain 70% CPU utilization, with replicas between 2-6. Use PodDisruptionBudget to ensure at least 1 pod is available during rolling updates.

### Network Tuning

Introduce retry policies in the HTTP clients with exponential backoff (e.g., 3 retries with initial delay of 50ms).
