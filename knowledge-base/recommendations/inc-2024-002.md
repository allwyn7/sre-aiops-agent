## Capacity Planning

### Current Utilization
HTTP 500 rate peaked at 91 errors/min, compared to a baseline of <0.2 errors/min. DB connection pool is operating at full capacity (HikariCP maxPoolSize=10).

### Scaling Thresholds
Scale up if error rate exceeds 10 errors/min or CPU usage crosses 70%. Increase database replicas if read latency increases by 20%.

### Resource Recommendations
Increase HikariCP maxPoolSize to 20. Add 1-2 more replicas of the service to handle production spikes. Set JVM heap to 1.5GB (current usage suggests ~1GB with 0.5GB headroom).

### Growth Projections
At 2x current load: 5% connection pool request latency may be observed. At 5x: DB replica saturation is likely. At 10x: latency for end-users will significantly increase, requiring horizontal scaling of both app and DB layers.

### Cost Considerations
Over-provision app pods to handle expected but rare traffic spikes or directly invest in auto-scaling and database read replicas to optimize for average load.

## Disaster Recovery Assessment

### Single Points of Failure
- Database schema consistency for critical queries.
- Single database instance as write leader.

### High Availability Recommendations
Add database read replicas for failover. Implement a canary deployment strategy to validate schema changes. Use feature flags to control microservice behavior during migrations.

### Failover Strategy
Enable PostgreSQL failover using a replica as a hot standby, synchronized with WAL logs. Adjust HikariCP config to route reads to replicas.

### RTO / RPO Assessment
RTO: 10 minutes (based on DB restore and redeployment). RPO: Up to 10 seconds of data loss, assuming regular write-ahead logs and backups.

### Backup Recommendations
Enable frequent backups of PostgreSQL using cron jobs with pg_dump. Test restore procedures monthly.

## Performance Tuning

### JVM Tuning
JAVA_OPTS: '-Xms1024m -Xmx1536m -XX:+UseG1GC -XX:InitiatingHeapOccupancyPercent=45'. Increase heap up to 2GB under sustained load.

### Spring Boot Configuration
application.yml: server.tomcat.max-threads=200 (increase concurrent request capacity), spring.datasource.hikari.maximum-pool-size=20

### JPA / Hibernate Tuning
hibernate.default_batch_fetch_size=50 (optimize for read queries), enable 2nd level cache for frequently accessed data.

### Database (PostgreSQL) Tuning
PostgreSQL: Set shared_buffers=256MB, effective_cache_size=768MB, work_mem=16MB for faster query processing. Add indexes for frequently queried fields like 'id'.

### Connection Pool (HikariCP) Tuning
HikariCP: Set maxPoolSize=20, idleTimeout=30000ms, maxLifetime=1800000ms (30 minutes). Enable connection leak detection (leakDetectionThreshold=5000ms).
