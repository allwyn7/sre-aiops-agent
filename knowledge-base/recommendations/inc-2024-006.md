# SRE Recommendations: INC-2024-006

> **Service:** `bookshop-srv` | **Severity:** P1 | **Date:** 2024-04-25

## Capacity Planning

### Current Utilization
Memory limits (512Mi) were exceeded with pods peaking at 510Mi during a traffic spike. CPU utilization reached 98% of the limit (490m out of 500m).

### Scaling Thresholds
Trigger horizontal scaling at CPU utilization above 70%. Monitor memory usage and consider scaling if usage exceeds 70% of the set limits.

### Resource Recommendations
Increase memory limits to 1Gi with requests set at 512Mi. Update HPA to handle up to 10 replicas, and maintain two minimum replicas during low traffic.

### Growth Projections
At 2x traffic, HPA may need to scale to 6 replicas with memory limits potentially reaching 768Mi per pod. At 5x, consider cluster node scaling. At 10x, distributed caching and additional service optimizations may be required.

### Cost Considerations
Increasing HPA replica count and memory reservation will increase infrastructure costs. Optimize application code, database queries, and caching layer to mitigate costs.

## Disaster Recovery Assessment

### Single Points of Failure
- HPA with maxReplicas=3
- Insufficient container memory limits

### High Availability Recommendations
Add a PodDisruptionBudget ensuring a minimum of 2 pods available. Add MixedInstancePolicy to the node pool to handle burst traffic and node failures.

### Failover Strategy
Configure HPA with higher maxReplicas to dynamically scale during traffic spikes. Use Kubernetes Cluster Autoscaler to increase available nodes when resource constraints arise.

### RTO / RPO Assessment
RTO = 30 minutes (time taken to update deployment specs and reallocate workloads). RPO = zero (no data loss occurred).

### Backup Recommendations
Maintain regular PostgreSQL backups with pg_dump or continuous archiving. Backup configurations should align with service SLAs to ensure quick recovery.
