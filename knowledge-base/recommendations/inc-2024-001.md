## Disaster Recovery Assessment

### Single Points of Failure
- Unbounded cache in BookService hampered the entire service, affecting all users and endpoints.
- Single JVM pod without sufficient replicas for failover during resource exhaustion.

### High Availability Recommendations
Increase the pod count using HPA. Use circuit breakers and request timeouts to shield users from cascading failures.

### Failover Strategy
Ensure a minimum of two replicas during normal operation. Use Istio service mesh to implement retries and fallbacks for critical API endpoints (e.g., /actuator/health).

### RTO / RPO Assessment
RTO: 30 minutes (with proper rollback), RPO: Near-zero since data was unaffected. Improve by enabling auto-detection through Prometheus alerts tied to CI/CD rollback automation.

### Backup Recommendations
Regular backup snapshots of PostgreSQL (e.g., daily). Versioned configurations in a source control repo for services. Use SAP BTP's Cloud Backup service to safeguard environments.

## Performance Tuning

### JVM Tuning
-Xms512m -Xmx1g to double memory limits. Evaluate switching to G1GC for lower pause times compared to default GC.

### Spring Boot Configuration
Add caching configurations to 'application.yml':
  spring:
    cache:
      type: caffeine
    caffeine:
      cache-spec: maximumSize=100,expireAfterWrite=10m
