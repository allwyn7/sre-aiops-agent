# Runbook: Cert-manager HTTP-01 Ingress Misconfiguration Recovery

> **Incident:** INC-2024-007 | **Service:** `bookshop-srv` | **Date:** 2024-04-28

**Trigger Pattern:** `acme: authorization failed: fetching http://<domain>/.well-known/acme-challenge/<token> returned status 404`

---

## 1. Detection

Monitor cert-manager logs and alert on failed renewal attempts: Look for patterns like 'authorization failed' or 'HTTP-01 verification returned 404'. Alert thresholds: consecutive failures > 3 or expiration less than 15 hours away.

## 2. Triage

1. Check cert-manager logs for HTTP-01 challenge failures. 2. Confirm if the TLS certificate has already expired (`kubectl describe certificate -n {namespace} {certificate-name}`). 3. Check if impacted endpoints are reachable over HTTPS or HTTP to assess the severity of the outage. Notify the on-call SRE and service owner immediately.

## 3. Diagnosis

1. Review the ingress definition using `kubectl describe ingress -n {namespace}` and confirm that the HTTP-01 solver route (/.well-known/acme-challenge) is configured. 2. Verify cert-manager logs for renewal attempts: `kubectl logs -n cert-manager deployment/cert-manager | grep 'Challenge failed'`. 3. Inspect the failed certificate: `kubectl describe certificate -n {namespace} {certificate-name}` and check status conditions for renewal errors.

## 4. Resolution

1. Correct the ingress configuration. Example ingress YAML provided in the remediation section. Validate that the ingressClass and challenge route are configured appropriately. Apply the fix using `kubectl apply -f ingress-cert-manager-fix.yaml`. 2. Delete the failed certificate resource using `kubectl delete certificate -n {namespace} {certificate-name}`. Cert-manager will recreate it and trigger renewal automatically. 3. Verify the new TLS certificate is issued using `kubectl describe certificate -n {namespace} {certificate-name}` and confirm that the 'Ready' condition is 'True'.

## 5. Verification

1. Confirm HTTPS traffic is restored by making HTTPS requests to the affected domain using a tool like `curl -v https://bookshop.example.com`. Check that the TLS handshake succeeds. 2. Verify the new certificate's expiration date and issuer (Let's Encrypt) using a browser or certificate inspection tool.

## 6. Prevention

1. Implement monitoring for certificates approaching expiration (`tls_cert_expiry_days < 7` alerts). 2. Configure periodic Kubernetes configuration validation in CI/CD pipelines to catch pre-deployment issues. 3. Enable cert-manager to use DNS-01 challenge instead of HTTP-01 for environments where ingress routing issues may occur. 4. Perform regular chaos engineering drills to verify the reliability of certificate renewal failovers.
