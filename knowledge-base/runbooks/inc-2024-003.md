# Runbook: N+1 Query Explosion in Hibernate/JPA

> **Incident:** INC-2024-003 | **Service:** `bookshop-srv` | **Date:** 2024-04-12

**Trigger Pattern:** `.*Servlet.service\(\) for servlet \[dispatcherServlet\] threw exception.*Unable to acquire JDBC Connection.*Connection pool exhausted.*`

---

## 1. Detection

Monitor logs for repeated patterns of 'Connection pool exhausted' or 'Unable to acquire JDBC Connection'. Additionally, observe metrics like DB connection pool usage (e.g., hikari_connection_pool_active), database query per request rates, and API response times to detect anomalies.

## 2. Triage

1. Verify application logs for error patterns indicating connection pool exhaustion or high query count.,2. Check metrics for hikari_connection_pool_active and hikari_connection_pool_waiting metrics.,3. Attempt to reproduce the issue with a test query in a non-production environment to confirm N+1.

## 3. Diagnosis

1. Look for recent changes in JPA entities or business logic surrounding affected endpoints.,2. Inspect entity relationship configurations, especially @ManyToOne or @OneToMany annotations, for recent changes in fetch types from LAZY to EAGER.,3. Use logs to identify SQL queries executed during the endpoint calls. A large number of SELECT queries for a single request typically indicates an N+1 problem.,4. Review the code responsible for the endpoint and verify patterns that may result in incremental fetching.

## 4. Resolution

1. If possible, revert the changes introduced by the offending PR to immediately restore stability to the service.,2. Modify the JPA entity to change the fetch type back to LAZY for the impacted fields.,3. Refactor the service class to use a single JOIN query (e.g., JPQL's JOIN FETCH) to fetch all required data in a single database query.,4. Test the fix in a staging environment to verify that the N+1 query problem is resolved.,5. Deploy the fix to production and monitor metrics (e.g., response times, database connection pool usage) to confirm resolution.

## 5. Verification

1. Confirm GET /api/books returns expected results without timing out or causing excessive query counts (>2).,2. Use APM tools or logs to verify no excessive database SELECT queries.,3. Ensure the hikari_connection_pool_active count stabilizes within configured limits and that the database connection pool is no longer exhausting.,4. Validate that the response times for the endpoint return to baseline levels (~92ms on average).

## 6. Prevention

1. Implement a JPA entity analysis CI pipeline to detect N+1 risks during pull request builds.,2. Include N+1 query detection as part of endpoint performance tests in staging environments.,3. Add monitoring to alert for abnormally high database query counts per request during load.,4. Regularly audit Hibernate configurations, especially fetch types, for all JPA entities.
