# Runbook: XSUAA JWT Audience Mismatch After xsappname Configuration Change

> **Incident:** INC-2024-004 | **Service:** `bookshop-srv` | **Date:** 2024-04-14

**Trigger Pattern:** `com.sap.cloud.security.xsuaa.token.TokenValidationException: JWT token audience '...' does not match configured xsappname '...'`

---

## 1. Detection

1. Application logs with error: `JWT token audience '...' does not match configured xsappname '...'`. 
2. Prometheus metrics: `http_401_rate_per_min > 10` or `error_rate_percent > 50%`. 
3. Application health check endpoint (`/actuator/health`) returning HTTP 401.

## 2. Triage

1. Check application logs for any sign of XSUAA token validation errors.
2. Identify recent code/config changes (check issue history linked to the application). 
3. Confirm if issues began manifesting post-deployment to production (check timestamp). 
4. Notify the service owner and on-call engineer immediately as this is a P1 incident affecting all endpoints.

## 3. Diagnosis

1. Confirm XSUAA audience mismatch: 
   - Search logs for recent errors indicating `AudienceValidationException` or `XSUAA token validation failed`.
   - Specifically, look for messages indicating that the `Actual audience` does not match the `Expected audience`.
2. Verify that the `xsappname` in `application.yml` or `mta.yaml` matches the expected audience in JWT tokens issued by the XSUAA server.
3. Confirm service binding changes for XSUAA configuration were applied locally by examining ENV variables (`VCAP_SERVICES`).
4. Check a recent PR history to find any xsappname/related changes that could have triggered the issue.

## 4. Resolution

1. Confirm the new correct `xsappname` from BTP subaccount settings.
2. Update both `application.yml` and the `mta.yaml` to match the new `xsappname`.
   - Update issuer URI and audience fields inside `application.yml`.
   - Update XSUAA `xsappname` in `mta.yaml` under resources.
3. Rebind the XSUAA service in the BTP production subaccount to generate updated service bindings.
   - Run `cf unbind-service bookshop-srv xsuaa`.
   - Rebind with `cf bind-service bookshop-srv xsuaa`.
4. Restart the Spring Boot application to apply updates and clear old tokens: `cf restart bookshop-srv`.
5. Validate fix:
   - Ensure `/actuator/health` endpoint returns HTTP 200.
   - Confirm successful API requests using a valid JWT issued after the service binding update.

## 5. Verification

1. Test `/api/books` endpoint using Postman or curl with a valid JWT issued after rebind and confirm HTTP 200.
2. Validate `/actuator/health` endpoint returns HTTP 200.
3. Monitor Prometheus metrics for reductions in `http_401_rate_per_min` to return to baseline (<2 errors/min).
4. Ensure error logs related to XSUAA audience validation have stopped.

## 6. Prevention

1. Add migration checklist steps in the release pipeline to validate `xsappname` changes are correct and aligned across configurations (e.g., Spring `application.yml`, `mta.yaml`).
2. Automate test cases to validate correct audience claim alignment via integration tests before deployment.
3. Add health check probes for token audience validation during startup and on `/actuator/health` endpoint.
