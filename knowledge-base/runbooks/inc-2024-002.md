# Runbook: JPA Entity Schema Drift Recovery

> **Incident:** INC-2024-002 | **Service:** `bookshop-srv` | **Date:** 2024-04-10

**Trigger Pattern:** `org.postgresql.util.PSQLException: ERROR: column does not exist`

---

## 1. Detection

Monitor application logs for 'SQLGrammarException' or 'PSQLException: column does not exist'. Track HTTP 5xx spikes on critical services including error rate anomalies above the baseline (e.g., 5% error rate threshold). Monitor health checks such as '/actuator/health' returning DOWN due to database issues.

## 2. Triage

1. Check the HTTP 5xx rate in Prometheus and look for affected endpoints.
2. Review logs for SQLGrammarException or PSQLException indicating missing columns.
3. Examine the recent deployment logs to identify code changes (e.g., PR/commit IDs).
4. Notify on-call SRE and assigned service owner.

## 3. Diagnosis

1. Run `kubectl logs -l app=bookshop-srv -n production | grep 'SQLGrammarException'` to extract the error logs.
2. Identify SQL query or column name causing the exception (e.g., column 'price_old').
3. Verify database schema by connecting to PostgreSQL and running `\\d+ <table_name>` or equivalent SQL queries to inspect schema.
4. Cross-reference affected schema with entity class definitions (e.g., check Book.java for 'priceOld').
5. Check recent PRs/commits for changes to affected schemas or entities.

## 4. Resolution

1. Prepare a Flyway migration (e.g., `db/migration/V2__drop_price_old_column.sql`) to drop the problematic column:
   ```sql
   ALTER TABLE book DROP COLUMN IF EXISTS price_old;
   ```
2. Commit the migration script to the repository and follow the pull request procedure.
3. Deploy the updated application with the database schema modification.
4. Monitor the service health using '/actuator/health' and ensure HTTP endpoints function correctly.

## 5. Verification

1. Ensure the '/actuator/health' endpoint reports 'UP' and database health is restored.
2. Verify logs for absence of 'SQLGrammarException' or 'PSQLException'.
3. Run test queries against fixed endpoints (e.g., `curl -X GET /api/books`) to confirm 200 responses.
4. Monitor error rates, ensuring return to baseline (e.g., http_5xx_rate_per_min < 0.2).

## 6. Prevention

1. Integrate database schema migration checks into the CI pipeline to prevent mismatches between code and database.
2. Enforce PR templates requiring schema modifications to include required migrations.
3. Implement alerts for exceptions such as SQLGrammarException in production logs.
4. Conduct post-deployment validation to confirm JPA entities and database schemas are in sync.
