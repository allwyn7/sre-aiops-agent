# Runbook: Java OOMError – Unbounded In-Memory Cache

> **Incident:** INC-2024-001 | **Service:** `bookshop-srv` | **Date:** 2024-04-08

**Trigger Pattern:** `java.lang.OutOfMemoryError: Java heap space.*HashMap.resize`

---

## 1. Detection

Monitor logs for 'java.lang.OutOfMemoryError', 'GC overhead limit exceeded', and rapid pod restarts. Metrics: 'jvm_heap_used_percent > 90' or 'pod_restarts_last_hour > 10'. Health check failures with /actuator/health returning DOWN.

## 2. Triage

1. Check the affected service logs for 'java.lang.OutOfMemoryError'.
2. Confirm pod restart frequency using 'kubectl get pods'.
3. Validate whether memory thresholds (e.g., jvm_heap_used_percent > 90) are exceeded in Prometheus metrics.
4. Notify the on-call SRE and service owner.

## 3. Diagnosis

1. Run 'kubectl logs <pod>' to inspect detailed logs for errors.
2. Check application code to identify unbounded data structures causing memory leaks.
3. Examine PRs and commits for recent changes linked to memory-intensive operations, such as caching or large object instantiations.
4. Check health endpoints (/actuator/health) and Prometheus for memory consumption patterns and GC activity.

## 4. Resolution

1. Rollback PR #48 immediately to remove the unbounded cache implementation.
   Command: 'kubectl rollout undo deployment bookshop-srv'.
2. Apply memory caps in Kubernetes YAML to control resource usage under load by increasing 'limits.memory' to 1Gi.
3. Update HPA metrics to scale pods before memory exhaustion (target CPU usage: ~75%).
4. Re-implement bounded caching strategy using LinkedHashMap or Caffeine library with eviction policies.
5. Redeploy the application using 'kubectl apply -f deployment.yml'.

## 5. Verification

1. Run 'kubectl get pods' to confirm the pod restart loop has subsided.
2. Verify logs no longer contain 'OutOfMemoryError' or GC warnings.
3. Test affected endpoints (e.g., /api/books) for response time improvements and error-free execution.
4. Monitor metrics such as 'jvm_heap_used_percent' to ensure it stays within acceptable thresholds (<70%).

## 6. Prevention

1. Add static code analysis to CI/CD pipelines to catch unbounded data structures.
2. Implement a load limit on caches (e.g., max-size, LRU policies).
3. Define Prometheus alerts for JVM memory and pod restart thresholds.
4. Conduct regular performance tests for new feature rollouts, especially caching changes.
