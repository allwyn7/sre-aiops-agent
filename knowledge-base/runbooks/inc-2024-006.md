# Runbook: Pod OOMKilled after Traffic Spike — Insufficient Memory Limits and HPA Configuration

> **Incident:** INC-2024-006 | **Service:** `bookshop-srv` | **Date:** 2024-04-25

**Trigger Pattern:** `.*OOMKilled: container .* terminated \(exit code 137\).*`

---

## 1. Detection

Look for 'OOMKilled: container ... terminated (exit code 137)' in Kubernetes logs or alerts indicating pod restarts, HPA maxed out, or high error rates (`http_5xx_rate_per_min`, `error_rate_percent`).

## 2. Triage

1. Check `kubectl get pods` for pods in CrashLoopBackOff or OOMKilled states. 2. Check HPA status with `kubectl get hpa` and `kubectl describe hpa`. 3. Validate node memory availability with `kubectl describe node` and check for memory pressure conditions.

## 3. Diagnosis

1. Run `kubectl get pods -n bookshop-prod` to list affected pods. 2. Inspect the specific pod with `kubectl describe pod -n bookshop-prod <pod-name>`. 3. Check HPA metrics with `kubectl describe hpa -n bookshop-prod bookshop-srv`. 4. Validate worker nodes' memory status using `kubectl describe node <node-name>`. 5. Use Prometheus queries (`rate(container_memory_working_set_bytes[5m])`) to analyze memory trends.

## 4. Resolution

1. Update memory limits in the deployment spec to 1Gi using the provided patch. 2. Increase HPA maxReplicas to 10 and adjust target CPU utilization to 70% using the provided HPA patch. 3. Apply the patches via `kubectl apply -f <path-to-patch.yaml>`. 4. If node memory pressure remains an issue, initiate node draining via `kubectl drain <node-name> --ignore-daemonsets --delete-emptydir-data` and redistribute pods.

## 5. Verification

1. Verify that pods are no longer in CrashLoopBackOff state using `kubectl get pods`. 2. Monitor memory and CPU usage in Prometheus to confirm stability. Use metrics such as `container_memory_working_set_bytes` and `container_cpu_usage_seconds_total`. 3. Ensure that HPA dynamically scales pods as per traffic requirements using `kubectl get hpa`. 4. Perform test requests against service endpoints to confirm HTTP 200 responses.

## 6. Prevention

1. Conduct regular capacity planning to ensure resource limits and HPA configurations align with expected traffic patterns. 2. Configure Prometheus alerts to detect high memory and CPU utilization early. 3. Implement load testing to simulate traffic spikes and fine-tune Kubernetes resource limits and HPA configurations. 4. Use a PodDisruptionBudget to ensure service availability during maintenance or node disruptions.
